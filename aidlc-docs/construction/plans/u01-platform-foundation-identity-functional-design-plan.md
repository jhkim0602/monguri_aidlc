# U01 Platform Foundation & Identity - Functional Design Plan

## 1. 목적
U01의 API Entry, Identity & Access, Notification 기반과 공통 감사·관측 계약을 기술 중립적으로 상세화한다. 계정·역할·프로필·세션·이메일 확인·비밀번호 재설정·삭제 전파·알림 상태의 엔터티, 상태 머신, 검증식, 오류 및 통합 계약을 정의한다.

## 2. 범위와 고정 경계
- 배포 경계: U01은 `Core API Service` 내부 Module과 공유 계약 패키지이며 별도 Service가 아니다.
- 소유 범위: 계정·역할·프로필·확인/재설정·세션·동의 참조·삭제 상태, 알림·전달 상태, 공통 감사·관측 메타데이터다.
- 비책임: 전문가 심사, 상담 Snapshot 권한, 타 Unit의 업무 상태, 런타임·DB·AWS·이메일·관측 제품 선택이다.
- 외부 계약: 리소스 중심 REST와 SSE 공통 형식, 상담 WebSocket 승격 계약을 제공하되 도메인 업무 규칙은 소유 Unit에 둔다.
- 데이터 경계: 다른 Module의 테이블을 직접 읽거나 수정하지 않고 Query, Port, 버전 이벤트만 사용한다.
- 확장 상태: Resiliency Baseline 활성화, PBT 부분 적용(PBT-02/03/07/08/09), Security Baseline 비활성화다.

- 답변은 기존 결정인 이메일/비밀번호 인증, 서버 측 fail-closed, U05 전문가 심사, U06 UI 소유, 삭제 전파, 서비스 내+이메일 알림과 모순되지 않아야 한다.
- 감사 필수 작업의 안전성과 월 99.9% 가용성 간 트레이드오프를 명시적으로 추적한다.

## 10. 콘텐츠 검증
- Mermaid/ASCII 다이어그램 없음: 별도 구문 검증 대상이 없다.
- Markdown 표, 인라인 코드, 링크와 질문 구조를 CommonMark 호환 형식으로 확인했다.
- 모든 질문은 의미 있는 선택지와 마지막 `X) 기타`, 빈 `[Answer]:`를 포함한다.
- 복잡 시각 요소가 없어 텍스트 대안은 필요하지 않다.
