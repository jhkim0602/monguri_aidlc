# U01 비즈니스 규칙

## 1. 목적
U01의 검증식, 상태 전이, 권한, 멱등성, 오류, 삭제, 알림 및 감사 규칙을 식별 가능한 규칙 ID로 정의한다. `MUST`는 위반 시 요청을 거부하거나 안전 상태를 유지해야 하는 필수 규칙이다.

## 2. 이메일과 계정 생성
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-001 | 이메일 비교 키는 앞뒤 공백을 제거하고 대소문자를 무시해 계산하며 원 입력 표시는 별도 보존한다. |
| U01-BR-002 | 공급자별 점·플러스 별칭을 임의 통합하지 않는다. |
| U01-BR-003 | 하나의 이메일 비교 키에는 삭제 완료 전 하나의 비종료 Account만 존재할 수 있다. |
| U01-BR-004 | 신규 계정은 `PENDING_EMAIL`이며 확인 전 보호 업무 기능을 사용할 수 없다. |
| U01-BR-005 | 같은 이메일의 확인 대기 계정 재가입은 새 계정을 만들지 않고 일반 메시지와 확인 재발급 흐름으로 수렴한다. |
| U01-BR-006 | 활성·정지·삭제 상태 이메일의 가입·재설정 응답은 계정 존재 여부를 드러내지 않는다. |
| U01-BR-007 | 가입 유형은 `JOB_SEEKER` 또는 `PROFESSIONAL_APPLICANT` 중 하나다. |

## 3. 이메일 확인과 재설정 Challenge
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-010 | 확인 Challenge는 발급 시점부터 24시간 동안 유효하다. |
| U01-BR-011 | 새 확인 Challenge 발급은 같은 목적의 기존 미사용 Challenge를 즉시 무효화한다. |
| U01-BR-012 | Challenge는 단회 사용이며 성공·만료·대체·계정 상태 변경 후 재사용할 수 없다. |
| U01-BR-013 | Challenge 검증은 목적, 대상 계정, 현재 세대, 만료, 사용 여부를 모두 확인한다. |
| U01-BR-014 | 확인 성공은 `PENDING_EMAIL`에서만 계정을 활성화하고 같은 요청의 재실행은 기존 성공 결과로 수렴한다. |
| U01-BR-015 | 비밀번호 재설정 성공은 해당 계정의 모든 Session을 폐기하고 모든 미사용 재설정 Challenge를 무효화한다. |
| U01-BR-016 | 비밀번호 변경도 요청 Session을 포함한 모든 Session을 폐기한다. |

## 4. 세션과 인증
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-020 | 이메일 확인 완료 및 허용 상태의 계정만 일반 Session을 발급받을 수 있다. |
| U01-BR-021 | 한 계정은 여러 활성 Session을 가질 수 있다. |
| U01-BR-022 | 일반 로그아웃은 요청 Session만 폐기하며 전체 로그아웃은 해당 계정의 모든 Session을 폐기한다. |
| U01-BR-023 | 폐기·만료·세대 불일치 Session은 ActorContext를 생성할 수 없다. |
| U01-BR-024 | 계정 정지, 삭제 요청, 비밀번호 변경·재설정은 기존 모든 Session을 즉시 폐기한다. |
| U01-BR-025 | 인증 실패 외부 메시지는 미등록, 비밀번호 오류, 정지, 미확인 여부를 구분하지 않는다. |
| U01-BR-026 | 인증 실패 횟수와 제한 정책의 수치는 NFR Design에서 정하되 계정 존재 노출을 증가시키지 않는다. |

## 5. 역할과 전문가 접근
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-030 | 구직 유형 가입자는 이메일 확인 후 `JOB_SEEKER` 역할을 얻는다. |
| U01-BR-031 | 전문가 유형 가입자는 이메일 확인 후 `PROFESSIONAL_APPLICANT` 제한 역할을 얻는다. |
| U01-BR-032 | 심사 대기 전문가는 신청 프로필 조회·수정과 심사 상태 조회만 허용된다. |
| U01-BR-033 | U05의 승인된 `ProfessionalVerified` 계약만 `PROFESSIONAL` 역할을 활성화할 수 있다. |
| U01-BR-034 | U05의 심사 거절은 제한 역할을 제거하고 계정을 활성 `JOB_SEEKER`로 전환한다. |
| U01-BR-035 | 거절 후 재신청 가능성은 U05 정책 결과를 따르며 U01이 임의 허용하지 않는다. |
| U01-BR-036 | 운영자 역할은 제한된 관리 절차만 부여·폐기할 수 있고 모든 결과를 감사한다. |
| U01-BR-037 | 역할 변경은 계정 상태와 충돌할 수 없으며 정지·삭제 상태가 역할 허용보다 우선한다. |
## 6. 프로필과 동시 수정
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-040 | 표시 이름은 필수이며 공백만 있는 값은 허용하지 않는다. |
| U01-BR-041 | 연락처 등 추가 개인정보는 선택이며 목적이 없는 필드를 강제 수집하지 않는다. |
| U01-BR-042 | 경력·전문 분야·업무 콘텐츠는 U01 Profile에 복제하지 않고 소유 Unit 참조를 사용한다. |
| U01-BR-043 | Profile 변경은 기대 버전을 사용하고 버전 불일치 시 저장하지 않는다. |
| U01-BR-044 | 충돌 응답은 사용자의 제출 내용을 폐기하지 않고 최신 버전과 비교 가능하게 한다. |

## 7. 권한 판정
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-050 | 권한 판정의 기본값은 거부다. |
| U01-BR-051 | 인증, Account 상태, Role Action, ResourcePolicy 중 하나라도 거부·누락·불명확하면 최종 거부한다. |
| U01-BR-052 | U01은 공통 역할·Action 정책을 판정하고 리소스 소유권·공유·기간은 원 소유 Unit의 정책 결과를 사용한다. |
| U01-BR-053 | `DELETION_GRACE` 제한 Session은 `CANCEL_DELETION` 외 Action을 허용하지 않는다. |
| U01-BR-054 | 권한 없는 리소스의 외부 응답은 존재 여부를 숨기며 내부 결정은 실제 거부 이유를 보존한다. |
| U01-BR-055 | 상담 Snapshot 기간·참가자 판정은 U04 결과 없이는 허용하지 않는다. |
| U01-BR-056 | AuthorizationDecision은 정책 버전, Actor, Action, 선택적 Resource, 이유와 상관관계를 포함한다. |

## 8. 계정 삭제
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-060 | 삭제 요청은 활성 Account 소유자의 현재 자격 증명 재검증과 영향 확인을 요구한다. |
| U01-BR-061 | 삭제 확정 즉시 `DELETION_GRACE`로 전이하고 모든 기존 Session을 폐기한다. |
| U01-BR-062 | 유예 기간은 삭제 확정 시각부터 정확히 7일이며 기간 중 일반 로그인·보호 요청을 거부한다. |
| U01-BR-063 | 유예 기간에는 삭제 전파 이벤트를 발행하지 않는다. |
| U01-BR-064 | 삭제 취소 제한 Session은 이메일·비밀번호 재검증으로만 만들고 현재 deletionGeneration의 취소 Action에만 사용할 수 있다. |
| U01-BR-065 | 취소 성공은 Account를 ACTIVE로 복원하지만 기존 Session을 복원하지 않으며 새 로그인을 요구한다. |
| U01-BR-066 | 유예 만료 시 `AccountDeletionRequested`를 AccountRef + deletionGeneration으로 발행한다. |
| U01-BR-067 | 삭제 소비자는 자기 데이터만 처리하고 generation별 중복을 같은 결과로 수렴시킨다. |
| U01-BR-068 | 모든 필수 Ack 전까지 Account를 삭제 완료로 표시하지 않는다. |
| U01-BR-069 | 일부 실패 시 Account는 `DELETION_PARTIAL_FAILURE`로 비활성 상태를 유지하고 같은 generation으로 재처리한다. |
| U01-BR-070 | 법적 보존 데이터는 목적·기간·접근 제한을 Ack에 표시하되 일반 Account를 활성화하지 않는다. |
| U01-BR-071 | 삭제 완료 후 최소 Tombstone은 중복 방지·감사·보존 목적에 필요한 정보만 가진다. |

## 9. 알림
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-080 | 보안·계정 범주의 서비스 내 알림과 이메일은 필수이며 사용자가 해제할 수 없다. |
| U01-BR-081 | 일정·상담 서비스 내 알림은 필수이며 이메일만 사용자가 해제할 수 있다. |
| U01-BR-082 | 서비스 내 알림은 이메일 작업보다 먼저 또는 같은 원자적 업무 의도로 기록한다. |
| U01-BR-083 | 이메일 전달 실패는 서비스 내 알림이나 원 업무 상태를 롤백하지 않는다. |
| U01-BR-084 | 알림 중복 키는 sourceRef, recipientRef, category, generation을 포함한다. |
| U01-BR-085 | 이메일 작업은 실행 직전 수신 가능 상태와 현재 주소 세대를 재검증한다. |
| U01-BR-086 | 반복 이메일 실패는 제한 재시도 후 격리하고 민감정보 없는 운영 신호를 만든다. |
| U01-BR-087 | 읽음 처리는 멱등하며 전달 상태를 변경하지 않는다. |

## 10. 감사와 관측
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-090 | 계정 삭제·취소, 비밀번호 변경·재설정, 전체 세션 폐기, 역할 변경, 정지는 필수 감사 Action이다. |
| U01-BR-091 | 필수 감사 Action은 업무 변경과 같은 원자적 경계에 감사 의도를 남기지 못하면 적용하지 않는다. |
| U01-BR-092 | 감사 이벤트는 행위자·대상·Action·시각·결과·결정 버전·상관관계를 포함한다. |
| U01-BR-093 | 비밀번호, Challenge/Session Token, 문서 원문, 전사문과 영상 내용은 감사·운영 로그에 포함하지 않는다. |
| U01-BR-094 | 감사 기록은 append-only 의미를 가지며 수정은 새 정정 이벤트로 표현한다. |
| U01-BR-095 | 일반 지표·로그·추적 전송 실패는 핵심 요청을 무기한 차단하지 않는다. |
| U01-BR-096 | 상관관계 ID는 API Entry에서 생성·검증하고 모든 내부 호출·이벤트에 전달한다. |
| U01-BR-097 | 관측 신호는 개인정보나 무제한 고카디널리티 값을 태그로 사용하지 않는다. |

## 11. API Entry와 계약
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-100 | API Entry는 계약 버전, 형식, 크기, 상관관계와 필요한 멱등 키를 업무 호출 전에 검증한다. |
| U01-BR-101 | API Entry는 업무 상태를 직접 변경하지 않고 소유 Command/Query로 위임한다. |
| U01-BR-102 | 보호 Route는 현재 ActorContext 없이는 업무 컴포넌트를 호출하지 않는다. |
| U01-BR-103 | SSE는 Operation 소유권을 재검증하고 단조 증가 event sequence와 재개 위치를 제공한다. |
| U01-BR-104 | Socket 승격은 U01 인증과 U04 참가자·기간 권한이 모두 허용일 때만 가능하다. |
| U01-BR-105 | 외부 오류는 안정된 code와 안전 메시지를 제공하고 내부 예외·비밀·계정 상태를 노출하지 않는다. |
| U01-BR-106 | 계약의 의미·버전은 데이터 소유 Unit이 소유하고 U01은 공통 envelope·패키징 규약만 관리한다. |

## 12. 멱등성과 이벤트 순서
| 규칙 ID | 규칙 |
|---|---|
| U01-BR-110 | 회원가입 재전송은 같은 이메일 비교 키에서 새 비종료 Account를 만들지 않는다. |
| U01-BR-111 | 확인·재설정 Challenge 성공 재전송은 같은 성공 결과 또는 이미 소비된 안전 결과로 수렴한다. |
| U01-BR-112 | 상태 변경 Command의 멱등 범위는 Actor, Command, IdempotencyKey와 대상 세대를 포함한다. |
| U01-BR-113 | AccountDeletionRequested와 Ack는 deletionGeneration보다 오래된 결과를 현재 상태에 적용하지 않는다. |
| U01-BR-114 | Professional review 결과는 ReviewVersion이 현재 반영 버전보다 새로울 때만 적용한다. |
| U01-BR-115 | 동일 EventId의 반복 소비는 한 번의 관찰 가능한 상태 변경만 만든다. |

## 13. 오류 분류
| 오류 코드 범주 | 내부 의미 | 외부 표현 | 상태 변경 |
|---|---|---|---|
| AUTHENTICATION_FAILED | 미등록·비밀번호 불일치·상태 제한 | 일반 인증 실패 | 없음 |
| CHALLENGE_INVALID | 만료·대체·사용·목적 불일치 | 링크 만료/무효 | 없음 |
| SESSION_INVALID | 만료·폐기·세대 불일치 | 재인증 필요 | 없음 |
| RESOURCE_HIDDEN | 역할·소유권·기간 거부 | 찾을 수 없음 | 없음 |
| VERSION_CONFLICT | 기대 버전 불일치 | 충돌 및 최신 버전 | 없음 |
| AUDIT_UNAVAILABLE | 필수 감사 의도 미수락 | 안전한 일시 실패 | 업무 변경 없음 |
| DELETION_PARTIAL | 필수 Unit Ack 실패 | 삭제 지연 상태 | Account 비활성 유지 |
| DELIVERY_DEGRADED | 이메일 재시도/격리 | 서비스 내 알림 유지 | 원 업무 유지 |
| CONTRACT_INVALID | 형식·버전·크기 오류 | 안전한 입력 오류 | 업무 호출 없음 |

## 14. 결정표: 보호 요청
| Session | Account | Role/Action | ResourcePolicy | 결과 |
|---|---|---|---|---|
| 무효/없음 | 무관 | 무관 | 무관 | 거부: 인증 필요 |
| 유효 | ACTIVE 아님 | 무관 | 무관 | 거부: 외부 안전 응답 |
| 유효 | ACTIVE | 거부/누락 | 무관 | 거부 |
| 유효 | ACTIVE | 허용 | 거부/누락 | 거부 |
| 유효 | ACTIVE | 허용 | 허용 또는 불필요 | 허용된 범위만 승인 |
| 삭제 취소 제한 | DELETION_GRACE | CANCEL_DELETION | 현재 generation | 삭제 취소만 허용 |
