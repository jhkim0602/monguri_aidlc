## 확정 설계 방향
1. **상위 구조**: 핵심 업무는 Epic 경계의 모듈러 모놀리스로 구성하고 AI·비동기 작업·면접 미디어·상담 WebRTC는 독립 실행 컴포넌트로 격리한다.
2. **업무 경계**: 승인된 9개 Epic을 상위 컴포넌트와 추적성의 기준으로 사용한다.
3. **클라이언트 인터페이스**: 리소스 중심 REST API를 기본으로 사용한다. 장시간 작업 상태는 SSE, 상담 채팅과 WebRTC 신호는 WebSocket으로 전달한다.
4. **장시간 작업**: AI 면접·영상 분석 같은 다단계 흐름은 워크플로 오케스트레이터가 관리하고 이메일·PDF·삭제 같은 단순 작업은 영속 상태와 큐 Worker로 처리한다.
5. **데이터 소유**: 하나의 관계형 데이터베이스에서 모듈별 스키마·테이블 소유권을 분리하고 대용량 미디어는 객체 저장소에 보관한다.
6. **AI 책임**: 공통 AI Gateway는 공급자 호출, 제한, 공통 감사 정보를 담당하고 프롬프트·근거·품질·사용자 결과는 각 업무 컴포넌트가 소유한다.
7. **실시간·미디어 경계**: 상담 WebRTC 통신 컴포넌트와 AI 면접 녹화·분석 컴포넌트를 분리한다.
8. **일관성**: 컴포넌트 내부는 강한 트랜잭션을 사용하고 컴포넌트 사이는 이벤트, 멱등 소비와 보상 작업으로 일관성을 유지한다.
9. **메서드 형태**: 외부 REST API는 `/jobs`, `/documents` 같은 자료 중심 계약을 사용하고 내부 서비스는 `RegisterJob`, `GenerateDocument`, `StartInterview` 같은 사용자 행동 중심 Command·Query와 명시적 입력·출력을 사용한다.
10. **과도한 복잡성 방지**: 독립 배포·워크플로·실시간 채널은 실제 격리 또는 장시간 처리 필요가 있는 경계에만 적용하고 단순 CRUD를 별도 서비스로 분리하지 않는다.
