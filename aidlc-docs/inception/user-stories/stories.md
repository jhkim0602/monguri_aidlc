# User Stories

## 1. 작성 규칙
- 각 도메인을 Epic으로 정의하고 요구사항 ID는 Epic에 연결한다.
- 각 스토리는 한 번의 사용자 결과와 하나의 핵심 검증에 집중한다.
- 인수 조건은 Given-When-Then 형식을 사용한다.
- 페르소나는 P-01 구직 사용자, P-02 전문가, P-03 운영자를 사용한다.

## 2. 공통 인수 규칙
다음 규칙은 모든 관련 스토리에 공통으로 적용한다.

- **CAR-01 인증**: 인증이 필요한 요청에서 유효한 세션이 없으면 접근을 거부하고 로그인 경로를 제공한다.
- **CAR-02 소유권**: 사용자는 자신이 소유하거나 명시적으로 공유받은 자료만 접근할 수 있다.
- **CAR-03 최소 권한**: 전문가와 운영자는 역할과 업무 목적에 필요한 최소 데이터만 볼 수 있다.
- **CAR-04 오류**: 실패 시 사용자가 다음 행동을 이해할 수 있는 한국어 메시지를 제공하고 입력 내용을 가능한 한 보존한다.
- **CAR-05 비동기 상태**: AI, 전사, 영상 분석, 이메일과 삭제 작업은 대기·진행·완료·실패·재시도 상태를 표시한다.
- **CAR-06 민감 로그**: 비밀번호, 토큰, 문서 원문, 전사문과 영상 내용은 평문 운영 로그에 기록하지 않는다.
- **CAR-07 감사**: 계정, 권한, 전문가 심사, 신고와 민감 자료 접근 상태 변경은 추적 가능한 감사 기록을 남긴다.
- **CAR-08 데이터 경계**: 민감 데이터 처리는 승인된 AWS 계정과 서울 리전 경계를 준수한다.
- **CAR-09 삭제**: 삭제 요청은 원본, 파생 데이터와 적용 가능한 백업 보존 정책에 전파되며 실패를 탐지한다.
- **CAR-10 사용성**: 주요 작업은 반응형 웹에서 상태, 오류와 완료 결과를 명확히 표시한다.

## 3. EP-01 계정과 접근
**목표**: 각 역할이 자신의 권한 범위에서 안전하게 플랫폼을 사용한다.

**요구사항 ID**: FR-AUTH-001~005, NFR-SEC-002~004

### US-01-01 이메일 회원가입
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 이메일과 비밀번호로 계정을 만들어 내 자료를 관리하고 싶다.

**Acceptance Criteria**
- **Given** 사용되지 않은 유효한 이메일과 정책에 맞는 비밀번호가 있을 때, **When** 회원가입을 완료하면, **Then** 이메일 확인이 필요한 개인 계정이 생성된다.

### US-01-02 이메일 확인
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 이메일 소유를 확인해 계정을 안전하게 활성화하고 싶다.

**Acceptance Criteria**
- **Given** 확인 대기 계정과 유효한 확인 링크가 있을 때, **When** 링크를 사용하면, **Then** 계정이 활성화되고 확인 상태가 기록된다.
### US-01-03 로그인과 로그아웃
**Persona**: P-01 구직 사용자, P-02 전문가, P-03 운영자

**Story**: 플랫폼 사용자로서 내 역할에 맞는 기능에 안전하게 접근하고 세션을 종료하고 싶다.

**Acceptance Criteria**
- **Given** 활성 계정의 올바른 자격 증명이 있을 때, **When** 로그인하면, **Then** 역할에 맞는 세션과 시작 화면이 제공된다.
- **Given** 로그인된 세션이 있을 때, **When** 로그아웃하면, **Then** 세션이 폐기되고 보호된 화면 접근이 거부된다.

### US-01-04 비밀번호 재설정
**Persona**: P-01 구직 사용자, P-02 전문가, P-03 운영자

**Story**: 플랫폼 사용자로서 비밀번호를 잊어도 이메일 확인을 통해 접근을 복구하고 싶다.

**Acceptance Criteria**
- **Given** 등록 이메일이 있을 때, **When** 재설정을 완료하면, **Then** 새 비밀번호가 적용되고 기존 재설정 링크와 세션이 무효화된다.

### US-01-05 프로필과 계정 삭제
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 개인정보를 수정하거나 계정을 삭제해 내 데이터를 통제하고 싶다.

**Acceptance Criteria**
- **Given** 인증된 개인 계정이 있을 때, **When** 프로필 변경을 저장하면, **Then** 검증된 값이 반영된다.
- **Given** 삭제 영향과 법적 보존 정보를 확인했을 때, **When** 계정 삭제를 확정하면, **Then** 계정이 비활성화되고 삭제 절차가 시작된다.

## 4. EP-02 채용공고와 지원 일정
**목표**: 구직 사용자가 공고 원문과 지원 일정을 한곳에서 관리한다.

**요구사항 ID**: FR-JOB-001~008

### US-02-01 공고 URL과 원문 등록
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고 URL과 원문을 직접 등록해 지원 준비를 시작하고 싶다.

**Acceptance Criteria**
- **Given** 출처 URL과 공고 원문이 있을 때, **When** 공고를 저장하면, **Then** 원문과 URL이 사용자 소유 공고로 보관되고 외부 페이지는 자동 수집되지 않는다.

### US-02-02 공고 정보 추출
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고 원문에서 핵심 정보를 자동 추출해 반복 입력을 줄이고 싶다.

**Acceptance Criteria**
- **Given** 공고 원문이 등록됐을 때, **When** 추출을 실행하면, **Then** 회사·직무·기술·자격·우대사항·마감일 후보가 검토 전 상태로 제시된다.

### US-02-03 추출 결과 검토
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 자동 추출된 공고 정보를 수정하고 확정해 정확한 기준을 만들고 싶다.

**Acceptance Criteria**
- **Given** 추출 후보가 있을 때, **When** 값을 수정하고 확정하면, **Then** 확정된 정보가 적합도 분석과 문서 생성의 기준이 된다.

### US-02-04 공고 원문 수정 이력
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고 원문을 수정하면서 이전 상태를 식별하고 싶다.

**Acceptance Criteria**
- **Given** 저장된 공고가 있을 때, **When** 원문을 수정하면, **Then** 최신 원문과 변경 시각을 확인할 수 있다.

### US-02-05 지원 상태 관리
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고별 지원 단계를 변경해 현재 진행 상황을 파악하고 싶다.

**Acceptance Criteria**
- **Given** 등록된 공고가 있을 때, **When** 허용된 지원 상태를 선택하면, **Then** 현재 상태와 변경 시각이 저장된다.

### US-02-06 전형 일정 관리
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 마감일과 전형 일정을 등록·수정·삭제해 준비 일정을 관리하고 싶다.

**Acceptance Criteria**
- **Given** 사용자 소유 공고가 있을 때, **When** 일정 작업을 완료하면, **Then** 공고와 연결된 최신 일정이 표시된다.

### US-02-07 일정 알림 수신
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 다가오는 지원과 상담 일정을 서비스 내 알림과 이메일로 받고 싶다.

**Acceptance Criteria**
- **Given** 알림 대상 일정이 있을 때, **When** 설정된 알림 시점이 되면, **Then** 서비스 내 알림이 생성되고 이메일 발송 결과가 추적된다.
## 5. EP-03 경험과 프로젝트
**목표**: 구직 사용자가 실제 경험을 구조화하고 여러 공고에서 재사용한다.

**요구사항 ID**: FR-EXP-001~006

### US-03-01 경험 등록
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 경험의 역할, 상황, 행동, 성과, 기술과 기간을 구조화해 저장하고 싶다.

**Acceptance Criteria**
- **Given** 경험 정보가 있을 때, **When** 필수 항목을 입력해 저장하면, **Then** 사용자 소유의 구조화된 경험이 생성된다.

### US-03-02 프로젝트 등록
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 프로젝트와 증빙 가능한 성과를 저장해 문서 근거로 사용하고 싶다.

**Acceptance Criteria**
- **Given** 프로젝트 정보와 사용자의 역할이 있을 때, **When** 프로젝트를 저장하면, **Then** 기간·기술·행동·성과가 구분된 프로젝트가 생성된다.

### US-03-03 경험 수정과 삭제
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 경험을 최신 상태로 수정하거나 더 이상 필요 없는 항목을 삭제하고 싶다.

**Acceptance Criteria**
- **Given** 사용자 소유 경험이 있을 때, **When** 수정 또는 삭제를 확정하면, **Then** 결과와 기존 문서에 미치는 영향이 표시된다.

### US-03-04 태그 관리
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 경험에 직무·역량·기술 태그를 붙여 필요한 경험을 쉽게 찾고 싶다.

**Acceptance Criteria**
- **Given** 저장된 경험이 있을 때, **When** 태그를 추가하거나 제거하면, **Then** 최신 태그로 검색과 연결 후보가 갱신된다.

### US-03-05 공고별 경험 재사용
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 하나의 경험을 여러 공고에 연결해 반복 작성을 줄이고 싶다.

**Acceptance Criteria**
- **Given** 경험과 여러 공고가 있을 때, **When** 경험을 공고에 연결하면, **Then** 원본 경험을 복제하지 않고 각 공고에서 참조할 수 있다.

### US-03-06 적합도 근거 검토
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고 요구사항과 경험의 적합도 및 연결 근거를 검토하고 싶다.

**Acceptance Criteria**
- **Given** 확정된 공고와 경험이 있을 때, **When** 적합도 분석을 실행하면, **Then** 요구사항별 관련 경험과 근거가 제시되고 사용자가 승인·수정·제외할 수 있다.

## 6. EP-04 근거 기반 AI 문서
**목표**: 구직 사용자가 실제 경험을 근거로 맞춤 문서를 생성·편집·확정한다.

**요구사항 ID**: FR-DOC-001~010, AI-001~005, AI-007~008

### US-04-01 문서 생성 근거 선택
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 문서에 사용할 공고와 경험을 직접 선택해 AI 입력을 통제하고 싶다.

**Acceptance Criteria**
- **Given** 확정된 공고와 연결 가능한 경험이 있을 때, **When** 생성 근거를 선택하면, **Then** 선택된 자료만 생성 요청에 포함된다.

### US-04-02 맞춤 이력서 초안 생성
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 공고와 선택 경험에 맞춘 이력서 초안을 생성하고 싶다.

**Acceptance Criteria**
- **Given** 공고와 경험 근거가 선택됐을 때, **When** 이력서 생성을 요청하면, **Then** 처리 상태가 표시되고 근거 범위 안의 초안이 생성된다.

### US-04-03 대화형 자기소개서 작성
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 질문별 대화를 통해 부족한 정보를 보완하고 자기소개서 초안을 만들고 싶다.

**Acceptance Criteria**
- **Given** 자기소개서 질문이 있을 때, **When** 대화형 안내에 답하면, **Then** 답변과 선택 경험을 근거로 질문별 초안이 갱신된다.

### US-04-04 문서 근거 목록 확인
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 AI 문서에 사용된 경험과 프로젝트 목록을 확인해 결과를 신뢰하고 싶다.

**Acceptance Criteria**
- **Given** 생성된 문서가 있을 때, **When** 근거 보기를 열면, **Then** 문서 생성에 사용된 경험·프로젝트 목록이 표시된다.
### US-04-05 근거 없는 사실 검토
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 근거가 부족한 제안을 확정 사실과 구분해 검토하고 싶다.

**Acceptance Criteria**
- **Given** 선택 근거로 확인할 수 없는 표현이 있을 때, **When** 결과를 표시하면, **Then** 해당 표현은 정보 보완이 필요한 제안으로 구분된다.

### US-04-06 초안 편집과 자동 저장
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 AI 초안을 직접 편집하고 자동 저장 상태를 확인하고 싶다.

**Acceptance Criteria**
- **Given** 편집 가능한 초안이 있을 때, **When** 내용을 변경하면, **Then** 변경이 자동 저장되고 최신 저장 상태가 표시된다.

### US-04-07 문서 확정 검토
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 AI 결과를 검토한 후에만 최종 문서로 확정하고 싶다.

**Acceptance Criteria**
- **Given** 생성 또는 편집된 초안이 있을 때, **When** 사용자가 검토 완료를 확인하면, **Then** 문서가 확정 상태로 전환된다.

### US-04-08 문서 버전 복원
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 이전 문서 버전을 조회하고 복원해 편집 실수를 되돌리고 싶다.

**Acceptance Criteria**
- **Given** 둘 이상의 문서 버전이 있을 때, **When** 이전 버전을 복원하면, **Then** 해당 내용이 새 최신 버전으로 생성된다.

### US-04-09 PDF 내보내기
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 확정된 문서를 PDF로 내려받아 지원에 사용하고 싶다.

**Acceptance Criteria**
- **Given** 확정된 문서가 있을 때, **When** PDF 내보내기를 실행하면, **Then** 화면 내용이 반영된 PDF 파일을 받을 수 있다.

### US-04-10 AI 결과 피드백 기록
**Persona**: P-01 구직 사용자

**Story**: 구직 사용자로서 AI 문장을 채택·수정·거절해 결과 품질 개선에 필요한 신호를 남기고 싶다.

### US-09-01 암호화와 비밀 관리
**Persona**: P-03 운영자

**Story**: 운영자로서 개인정보와 비밀값이 안전하게 저장·전송되어 노출 위험을 줄이고 싶다.

**Acceptance Criteria**
- **Given** 민감 데이터와 애플리케이션 비밀이 있을 때, **When** 시스템이 저장하거나 전송하면, **Then** 승인된 암호화와 관리형 비밀 저장소가 사용된다.

### US-09-02 AI·영상 비동기 처리 상태
**Persona**: P-03 운영자

**Story**: 운영자로서 장시간 AI·전사·영상 작업의 상태와 실패를 확인해 사용자 요청을 복구하고 싶다.

**Acceptance Criteria**
- **Given** 비동기 작업이 실행될 때, **When** 운영 상태를 조회하면, **Then** 작업 유형·상태·재시도 가능 여부·민감정보가 제거된 오류가 표시된다.

### US-09-03 월 99.9% 상태 관측
**Persona**: P-03 운영자

**Story**: 운영자로서 지연시간, 오류율, 처리량과 포화도를 확인해 월 99.9% 목표를 추적하고 싶다.

**Acceptance Criteria**
- **Given** 서비스가 운영 중일 때, **When** 상태 대시보드를 열면, **Then** 핵심 컴포넌트 지표와 가용성 목표 대비 상태가 표시된다.

### US-09-04 중앙 로그와 추적
**Persona**: P-03 운영자

**Story**: 운영자로서 민감정보가 제거된 로그와 요청 추적으로 다중 컴포넌트 오류를 조사하고 싶다.

**Acceptance Criteria**
- **Given** 사용자 요청이 여러 컴포넌트를 통과할 때, **When** 오류를 조사하면, **Then** 공통 요청 식별자로 구조화 로그와 추적 정보를 연결할 수 있다.

### US-09-05 영상 자동 삭제 감시
**Persona**: P-03 운영자

**Story**: 운영자로서 7일 영상 삭제 실패를 감지하고 재처리해 보존 정책을 지키고 싶다.

**Acceptance Criteria**
- **Given** 삭제 만료 영상이 있을 때, **When** 자동 삭제가 실패하면, **Then** 대상이 재시도 상태로 전환되고 민감정보 없는 경고가 생성된다.
### US-09-06 다중 가용영역 운영
**Persona**: P-03 운영자

**Story**: 운영자로서 서울 리전의 한 가용영역에 장애가 나도 핵심 서비스가 계속 동작하길 원한다.

**Acceptance Criteria**
- **Given** 다중 가용영역 구성이 있을 때, **When** 하나의 가용영역이 비정상이 되면, **Then** 수동 제어 작업 없이 정상 영역이 핵심 트래픽을 처리한다.

### US-09-07 백업과 복원 검증
**Persona**: P-03 운영자

**Story**: 운영자로서 암호화된 자동 백업을 검증해 수 시간 수준의 Backup and Restore 목표를 지원하고 싶다.

**Acceptance Criteria**
- **Given** 영속 데이터 백업이 있을 때, **When** 정기 복원 검증을 수행하면, **Then** 복원 성공 여부와 RTO·RPO 범위 결과가 기록된다.

### US-09-08 의존성 장애 격리
**Persona**: P-03 운영자

**Story**: 운영자로서 AI, 이메일 또는 저장소 장애가 전체 서비스로 확산되지 않게 하고 싶다.

**Acceptance Criteria**
- **Given** 외부 또는 관리형 의존성이 지연·실패할 때, **When** 호출 제한에 도달하면, **Then** 타임아웃·제한 재시도·실패 격리와 단계적 저하가 적용된다.

### US-09-09 배포와 이전 버전 복구
**Persona**: P-03 운영자

**Story**: 운영자로서 검증된 버전을 직접 배포하고 실패 시 이전 Git·아티팩트·IaC 버전으로 복구하고 싶다.

**Acceptance Criteria**
- **Given** 자동 검증을 통과한 버전과 이전 정상 버전이 있을 때, **When** 배포 실패가 확인되면, **Then** 이전에 고정된 버전을 재배포할 수 있다.

### US-09-10 경량 장애 대응 기록
**Persona**: P-03 운영자

**Story**: 운영자로서 장애의 알림, 원인, 복구와 재발 방지 조치를 간단히 기록하고 싶다.

**Acceptance Criteria**
- **Given** 사용자 영향 장애가 발생했을 때, **When** 복구가 완료되면, **Then** 영향·원인·복구·후속 조치를 포함한 경량 사후 기록이 생성된다.

### US-09-11 반응형 핵심 흐름
**Persona**: P-01 구직 사용자, P-02 전문가, P-03 운영자

**Story**: 플랫폼 사용자로서 데스크톱과 주요 모바일 브라우저 폭에서 핵심 작업을 완료하고 싶다.

**Acceptance Criteria**
- **Given** 지원되는 브라우저와 화면 폭이 있을 때, **When** 역할별 핵심 흐름을 수행하면, **Then** 정보나 필수 조작이 화면 밖에 숨지 않고 완료 가능하다.

### US-09-12 속성 기반 품질 검증
**Persona**: P-03 운영자

**Story**: 운영자로서 직렬화 왕복과 핵심 불변식을 다양한 생성 입력으로 검증해 숨은 데이터 오류를 줄이고 싶다.

**Acceptance Criteria**
- **Given** 선택된 언어의 PBT 프레임워크와 도메인 생성기가 있을 때, **When** CI에서 속성 테스트를 실행하면, **Then** 왕복·불변식이 검증되고 실패 시 축소된 입력과 재현 시드가 기록된다.

## 13. INVEST 검토
| 기준 | 적용 결과 |
|---|---|
| Independent | 스토리는 한 사용자 결과를 중심으로 분리했으며 공통 규칙은 중복을 줄인다. |
| Negotiable | 구현 기술 대신 사용자 목표와 관찰 가능한 결과를 정의한다. |
| Valuable | 모든 스토리는 구직 사용자, 전문가 또는 운영자의 명시적 가치를 포함한다. |
| Estimable | 도메인 Epic과 작은 결과 단위로 범위가 식별 가능하다. |
| Small | 각 스토리는 하나의 핵심 행동 또는 운영 결과에 집중한다. |
| Testable | 모든 스토리에 Given-When-Then 인수 조건이 있다. |

## 14. 확장 기능 준수

### Resiliency Baseline
- **준수**: 중요 사용자 흐름, 다중 가용영역, 관측성, 백업·복원, 의존성 격리, 롤백 및 장애 대응 가치를 EP-09에 반영했다.
- **N/A**: 구체적 인프라 설정, 헬스 체크, 확장 한도, 복구 Runbook과 복원력 실험은 후속 설계 단계에서 검증한다.
- **차단 발견 사항**: 없음.

### Property-Based Testing
- **준수**: 부분 적용 대상인 왕복, 불변식, 도메인 생성기, 축소·시드 재현 및 프레임워크 요구를 US-09-12에 반영했다.
- **N/A**: 실제 프레임워크 구성과 테스트 구현은 NFR Requirements 및 Code Generation 단계에서 검증한다.
- **차단 발견 사항**: 없음.

### Security Baseline
사용자 결정에 따라 비활성화했다. 프로젝트 고유의 인증, 최소 권한, 암호화, 감사와 삭제 조건은 공통 인수 규칙과 관련 Epic에 유지했다.
